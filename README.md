# Locus

A desktop application that parses a codebase and renders it as an interactive code graph. Point it at any Python, JavaScript, or Java project and it produces a navigable map of every file, class, function, method, variable, and import ‚Äî plus the relationships between them.

Built with a Python analysis backend and a React + Electron frontend. The two halves communicate through a single `graph.json` file.

---

## Features

- **Multi-language support** ‚Äî Python, JavaScript (`.js`, `.jsx`, `.mjs`, `.cjs`), and Java, all auto-detected per file by extension. Mixed-language projects work out of the box.
- **Structural view** ‚Äî files are rendered as containers, classes nest inside files, methods nest inside classes. The layout is computed automatically from the containment hierarchy.
- **Semantic clustering view** ‚Äî optional ML pipeline groups related classes and functions into labelled domain clusters using CodeBERT embeddings, UMAP dimensionality reduction, and HDBSCAN. Cluster names and descriptions are generated by Claude.
- **Interactive canvas** ‚Äî pan and zoom freely, drag individual nodes to rearrange, and use the node-spacing slider to breathe room into the layout.
- **Level-of-detail rendering** ‚Äî containers that are too small on screen collapse to a simplified view; their children are unloaded from the DOM until you zoom back in.
- **Click-through to source** ‚Äî selecting a node shows its file path, line range, and docstring. Clicking "Open in VS Code" jumps directly to the definition.
- **Edge types** ‚Äî imports, inheritance, calls, and (in semantic mode) domain membership are all rendered with distinct colours and arrow styles.
- **Colour customisation** ‚Äî every node type has an editable colour; changes are saved per visualization.
- **Save & restore** ‚Äî the graph, node positions, view settings, and colour overrides are persisted automatically after each analysis. The home screen shows a recents list; re-opening a saved visualization is instant.
- **Progress tracking** ‚Äî a real-time progress bar streams live updates from the backend as it parses each file.

### Analysis pipeline (backend)

The backend runs three sequential passes over the source tree:

**Pass 1 ‚Äî Parse.** Every supported source file is parsed with its language's tree-sitter grammar. Emits `file`, `class`, `function`, `method`, `variable`, and `import` nodes, plus `contains` edges that encode the containment hierarchy.

**Pass 2 ‚Äî Resolve imports.** Import statements are matched against the local file tree:
- Python: dotted module paths resolved against the file system, with same-directory and short-name fallbacks.
- JavaScript: relative specifiers (`./x`, `../y`) resolved with extension and `index.js` inference. Bare specifiers (npm packages) become external import nodes.
- Java: fully-qualified class names and package wildcards resolved via a package ‚Üí file map built in pass 1.

Imports that resolve to local files become `imports` edges between nodes. Unresolvable ones become standalone `import_module` / `import_entity` nodes.

**Pass 3 ‚Äî Resolve calls.** Call expressions extracted during parsing are matched against the entity lookup table ‚Äî same-file first, then cross-file when the name is unambiguous. Resolved calls become `calls` edges; unresolvable ones are dropped.

### Semantic clustering pipeline (optional)

Enabled with the "Semantic Clustering" toggle on the home screen (or `--cluster` flag via CLI). Runs after the three base passes:

1. **Extract** ‚Äî builds a text representation for each class and function (including child method bodies for richer context).
2. **Embed** ‚Äî encodes each entity with [CodeBERT](https://huggingface.co/microsoft/codebert-base) to a 768-dimensional vector.
3. **Cluster** ‚Äî UMAP reduces to 10 dimensions; HDBSCAN finds clusters automatically. Micro-clusters are merged for onboarding-friendly granularity. A second UMAP pass produces 2D layout coordinates.
4. **Label** ‚Äî the Claude API generates a human-readable domain name and short description for each cluster.
5. **Enrich** ‚Äî cluster nodes and `belongs_to` edges are appended to `graph.json`. UMAP coordinates are attached to each entity node as layout hints.

### Node types

| Type | Description |
|---|---|
| `file` | Source file; top-level container |
| `class` | Class definition; child of a file |
| `function` | Module-level function; child of a file |
| `method` | Method; child of a class |
| `variable` | Module-level or class-level variable |
| `import_module` | External package / module imported but not found locally |
| `import_entity` | Specific name imported from an external module |
| `cluster` | Semantic domain group (only present when clustering is enabled) |

### Edge types

| Type | Description |
|---|---|
| `imports` | File or entity imports another local node or an external import node |
| `inherits` | Class extends another class |
| `calls` | Function or method calls another function or method |
| `belongs_to` | Entity belongs to a semantic cluster |

## Setup

### Requirements

- Python 3.9+
- Node.js 18+
- npm

### Backend

```bash
cd locus-backend
pip install -r requirements.txt
```

This installs tree-sitter and the Python, JavaScript, and Java grammars. The semantic clustering pipeline additionally requires `torch`, `transformers`, `umap-learn`, and `hdbscan`, which are pulled in automatically when `--cluster` is first used.

### Frontend

```bash
cd locus-frontend
npm install
```

---

## Usage

### Running the desktop app

```bash
cd locus-frontend
npm run electron:dev
```

Enter a path to any local codebase in the home screen and click **Visualize**. Enable **Semantic Clustering** before clicking if you want domain-cluster overlays (requires `ANTHROPIC_API_KEY` and `HF_TOKEN` ‚Äî see [API keys](#api-keys)).

### Running the backend alone

```bash
cd locus-backend

# Analyze a project; output goes to output/graph.json
python analyze.py path/to/project

# Custom output path
python analyze.py path/to/project --output custom/path.json

# Analyze with semantic clustering
python analyze.py path/to/project --cluster

# Use the bundled sample projects
python analyze.py sample_projects
```

### Standalone clustering CLI

```bash
cd locus-backend

python cluster.py path/to/project
python cluster.py path/to/project --output results.json
python cluster.py path/to/project --min-cluster-size 6
python cluster.py path/to/project --no-labels   # skip Claude API, use entity names
```

---

## API keys

Semantic clustering requires two keys. Copy `.env.example` to `.env` inside `locus-backend/` and fill them in:

```bash
cp locus-backend/.env.example locus-backend/.env
```

```
ANTHROPIC_API_KEY=sk-ant-...   # Required for Claude cluster labelling
HF_TOKEN=hf_...                # Required for faster CodeBERT download from HuggingFace
```

Keys set via the in-app **Settings** screen (gear icon on the home screen) take precedence over the `.env` file.

## Development

### Web-only frontend (no Electron)

```bash
cd locus-frontend
npm run dev    # Vite at http://localhost:5173
```

`window.electronAPI` is not available in this mode. Load a graph by calling `useGraphStore.getState().setGraph(data)` from the browser console with a parsed `graph.json` object, or temporarily wire up a file input.

### Production preview

```bash
cd locus-frontend
npm run electron:preview   # Builds Vite then launches Electron
```

### Saves

Saves live in the Electron `userData` directory (`AppData/Roaming/locus` on Windows). Each codebase gets one save file keyed on a SHA-1 of its project path, so re-analyzing the same directory overwrites the previous save. An `index.json` holds lightweight metadata for the recents list.
